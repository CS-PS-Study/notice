## 진행 방식

### 과제 제출
- **책에 수록된 문제들을 풀어서 GitHub에 Pull Request 열기**
- **제출 기한**: 매주 **화요일 23:59**까지
- 모르는 문제는 **주석만이라도 작성해서 제출**, 이후 추가 수정 가능
- **미제출 벌금 및 패널티 없음**
  - 단, 제출이 불가능한 경우 **사전에 공유하기**

### 알고리즘 공부
- **기록 방식**: 각자 자유로운 형식과 분량으로 정리 (기본 매주, 사정 있을 시 조정 가능)
- **참고 자료**: 유튜브 강의 영상 활용 가능 (노션 참고)
- **코드 리뷰**: 매주 화요일 23:59까지 Pull Request 제출 → 다음 주 동안 서로 코드 리뷰
  - 코드 리뷰는 짧게 작성 가능 (좋았던 점, 개선할 점 등)
  - 우선 GitHub PR로 진행, 필요시 디스코드 회의 고려

### 문제 풀이
- **저장소 구조**: 각자 본인 이름으로 레포지토리 생성
```
  your-name/
  ├── week1/              # 1주차 문제 (백준번호.py)
  │   ├── 1000.py
  │   ├── 1526.py
  │   └── 65346.py 
  ├── week2/              # 2주차 문제 (백준번호.py)
  │   ├── 1151.py
  │   └── 1_2462.py       # 이전 주차 추가 문제 (주차_문제번호.py)
  └── ...
```
- **제출 및 리뷰**: 
  1. 매주 화요일 23:59까지 Pull Request 제출
  2. 다음 주 동안 코드 리뷰 진행
  3. 피드백을 반영/미반영 할지는 자유롭게 선택
  4. 모든 리뷰 완료 후 각자 Merge

- **(선택)** 이전 주차 알고리즘 복습 문제 2~3개 추가 풀이 (난이도: 상/중/하 혼합)

## 코딩 컨벤션

### 네이밍 규칙 (Python)
```python
# 변수: 스네이크 표기법 (snake_case)
hello_world = 10

# 함수: 카멜 표기법 (camelCase)
def helloWorld():
    pass

# 상수: 대문자와 언더스코어
HELLO_WORLD = 100
```

### 클린 코드 작성법
- **전역변수 최대한 사용 ❌**
- **함수의 용도는 명확하게**, 가능한 메인은 짧게
- **인덴테이션은 최대 3-depth까지** (삼중 for문까지 권장)
- **구현이 까다로운 부분은 주석으로 보충 설명**

## 효과적인 학습 방법

### 1. 시간 제한 설정
- 문제 풀이 시 시간을 정해두고 시작
- 시간 초과 시 답 확인 후, **다음날 스스로 다시 풀어보기**

### 2. 논리적 설명 연습
- 문제 풀이 과정을 **말로 설명**
- 논리적으로 맞지 않는 부분 점검

## 코딩 테스트 효율적 공부법

### 1. 시간/공간 복잡도 분석 습관화
- 코드 작성 전 예상 시간복잡도 계산
- 문제의 입력 크기에 맞는 알고리즘 선택
- `O(N)`, `O(N log N)`, `O(N²)` 등을 의식하며 코딩

### 2. 문제 유형별 패턴 익히기
- 유형별로 자주 사용되는 알고리즘과 자료구조 파악
- 비슷한 유형의 문제를 반복해서 풀며 패턴 내재화
- 예: 투 포인터, 슬라이딩 윈도우, DP, 그리디 등

### 3. 자주 사용하는 라이브러리/함수 숙지
```python
# Python 필수 라이브러리
from collections import deque, defaultdict, Counter
from heapq import heappush, heappop
import bisect
import itertools
```

### 4. 예외 케이스 철저히 테스트
- 최소/최대 입력값
- 빈 배열, 단일 원소
- 중복 값, 음수, 0
- 경계값(boundary) 케이스

### 5. 반복 학습의 중요성
- 한 번 푼 문제도 일정 시간 후 다시 풀어보기
- 특히 틀렸던 문제는 **왜 틀렸는지 명확히 알기**

### 6. 코드 최적화보다 정확성 우선
- 처음엔 완벽하지 않아도 **동작하는 코드** 작성
- 동작 확인 후 시간/공간 최적화 진행
- 과도한 최적화는 오히려 버그 발생 가능

### 7. 문제 해결 프로세스 정립
```
1. 문제 정확히 이해 (입력/출력 형식, 제약 조건)
2. 접근 방법 고민 (브루트포스 → 최적화)
3. 의사코드(pseudo code) 작성
4. 실제 코드 구현
5. 테스트 케이스 검증
6. 시간/공간 복잡도 분석
```
